# Перенос кабинета на сервер

Как поднять кабинет на собственном сервере (Linux). База остаётся в облаке Supabase —
её переносить не нужно, сервер просто к ней подключается.

## Что переносим, а что нет

- **Код** — забирается из GitHub: `git@github.com:mentori1/natalivm.git`.
- **База данных** — НЕ переносим. Она в Supabase (облако), доступна откуда угодно по `DATABASE_URL`.
- **Секреты** (`.env`, `.env.local`) — их нет в гите. На сервере создаём заново,
  значения берём из локальных файлов на этом компьютере.
- `node_modules`, `.next`, `dev.db` — переносить не нужно: первое и второе пересоздаются
  командами ниже, третье (старый SQLite) больше не используется.

## Что нужно на сервере

- Node.js **20+** и npm
- git
- (для красивого адреса и https) nginx как обратный прокси

## Шаги

1. Забрать код:
   ```bash
   git clone git@github.com:mentori1/natalivm.git
   cd natalivm
   ```

2. Создать файл `.env` со строкой подключения к базе
   (взять из локального `.env` или из Supabase → **Connect → Session pooler**):
   ```
   DATABASE_URL="postgresql://...pooler.supabase.com:5432/postgres"
   ```

3. Создать файл `.env.local` со входом (значения — из локального `.env.local`):
   ```
   APP_PASSWORD="ваш-пароль"
   AUTH_SECRET="длинная-случайная-строка"
   ```

4. Установить зависимости и собрать:
   ```bash
   npm install
   npm run build
   ```

5. Запустить:
   ```bash
   npm run start        # поднимется на http://СЕРВЕР:3000
   ```

## Чтобы кабинет работал постоянно

`npm run start` завершится, когда закроете сессию терминала. Чтобы кабинет работал всегда,
держите его под менеджером процессов — например **pm2**:

```bash
npm i -g pm2
pm2 start "npm run start" --name kabinet
pm2 save
pm2 startup          # автозапуск после перезагрузки сервера
```

## Домен и HTTPS

Чтобы открывать по адресу (а не по `IP:3000`) и по https — поставьте nginx, который
проксирует ваш домен на `localhost:3000`, и выпустите сертификат (Let's Encrypt / certbot).
Подключение к Supabase по `DATABASE_URL` при этом не меняется.

## Про подключения к базе

На Vercel (serverless) была проблема: множество мелких инстансов разом открывали коннекты
и упирались в лимит Supabase (15). Поэтому в [`src/lib/db.ts`](src/lib/db.ts) стоит `max: 1`.
На обычном сервере это уже не проблема — там один постоянный процесс с одним пулом.
Менять ничего не нужно, всё будет работать.

## Если остаётесь на Vercel

Vercel тоже продолжает работать: достаточно `git push` в ветку `main` — деплой соберётся сам.
Переменные окружения (`DATABASE_URL`, `APP_PASSWORD`, `AUTH_SECRET`) задаются в настройках
проекта на Vercel.

## После переноса — проверить

1. Открывается страница входа, пароль (`APP_PASSWORD`) пускает внутрь.
2. На дашборде видны реальные клиенты — значит, база подключилась.
3. Добавить тестовый расход и удалить — значит, запись в базу работает.
